# 自動販売機シミュレータ 設計ドキュメント

---

## 1. 自分で決めた要件（扱う商品、釣銭ルールなど）とその理由

### 扱う商品
本アプリでは以下の商品を扱うように設計した。

- 水（100円・冷たい）
- お茶（120円・冷たい／温かい）
- コーヒー（130円・冷たい）
- コーンポタージュ（140円・温かい）
- ココア（150円・温かい）

### 釣銭・投入金額ルール
- 投入可能金額：10円〜1000円まで
- 1000円を超える投入は禁止
- 返金ボタンで全額返却可能

### 在庫ルール
- 各商品は初期在庫を「5」に設定
- 購入ごとに1ずつ減少
- 在庫が0になった場合は「売り切れ」と表示し購入不可

### この要件にした理由
- 実際の自動販売機に近いものを再現するため
- 扱う商品数を限定し、商品管理や購入処理などの各関数のロジックの理解を優先したため
- 金額上限を設けることで「現実的な制約」を再現するため
- 在庫管理を入れることで状態変化の学習を目的としたため

---

## 2. データ構造と関数分割の方針（迷った点と選んだ理由）

### データ構造の設計
- 商品データは配列で管理し、各商品はオブジェクトで表現した。
- 飲み物温度の表示文言はオブジェクトで一元管理した。
```js
const TEMP = {
  HOT: "あったか〜い",
  COLD: "つめた〜い"
};
const products = [
{
  name: "水",
  price: 100,
  temp: TEMP.COLD,
  stock: 5
}
];
```

### 迷った点
- 商品オブジェクトを個別に定義してから配列に格納するか、最初から配列内に直接記述するか

### 採用理由
- 配列にすることで商品一覧表示が簡単になるため
- オブジェクト形式にすることで商品ごとの状態管理がしやすいため
- tempの値を定数として管理することで、表示文言の変更を容易にし、文字列の入力ミスを防ぐため

---

### 関数分割の方針
機能ごとに関数を分割し、役割を明確にした。
- renderProducts()：商品一覧を画面に表示
- insertMoney(amount)：金額の投入処理
- buyProduct(index)：購入処理
- addLog(message)：ログ追加処理

### 迷った点
- 在庫チェックや投入金額チェックなど、どこまでを関数にまとめるべきか

### 採用理由
- 在庫チェックや投入金額チェックは購入処理と密接に関係するため、buyProduct()内で条件分岐として実装した
- 機能ごとに関数を分けることで可読性を上げるため
- 各関数の役割を明確にし、バグの特定をしやすくするため
- 後から機能の追加・編集・削除がしやすくなるため
